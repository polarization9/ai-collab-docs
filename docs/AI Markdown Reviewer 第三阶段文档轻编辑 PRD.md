# AI Markdown Reviewer 第三阶段文档轻编辑 PRD

## 1. 文档信息

- 产品名称：AI Markdown Reviewer
- 阶段名称：第三阶段，文档轻编辑闭环
- 文档日期：2026-06-03
- 目标版本：v0.3.0
- 当前状态：需求草案

## 2. 阶段目标

第三阶段从“可阅读、可批注、可被 Agent 处理批注的 Markdown 审阅器”升级为“用户和 Agent 都可以直接修改 Markdown 正文的本地文档协作工具”。

目标体验：

> 用户打开一份本地 Markdown 文档，在阅读和批注之后，可以切换到编辑模式直接修改整篇 Markdown 源码；也可以让 Codex / Agent 根据明确的修改型批注修改正文。修改保存后，文档渲染刷新，批注尽量重新定位到修改后的文本上，Agent 可以追加回复并根据处理结果建议性地更新批注状态。

第三阶段不做完整富文本编辑器，也不做块级编辑。产品优先提供一个稳定、可撤销、可保存、可被 Agent 调用的整篇 Markdown 轻编辑能力。

## 3. 用户价值

- 用户不需要跳出阅读器到外部编辑器，即可完成正文小改和整篇整理。
- 用户可以用 Markdown 原文完成标题、段落、列表、表格、代码块和 Mermaid 的统一编辑。
- 批注、回复和正文修改形成闭环，减少“批注在这里、修改在别处”的割裂。
- Agent 可以直接完成明确的修改型批注，并把处理结果回写到批注线程。
- 文档修改后，批注锚点不会轻易丢失，后续审阅仍然有上下文。

## 4. 核心概念

### 4.1 阅读模式

阅读模式是当前默认状态，用于浏览 Markdown 渲染结果、查看目录、阅读 Mermaid 图表、查看表格和处理批注。

### 4.2 编辑模式

编辑模式是整篇 Markdown 源码编辑状态。用户看到并编辑当前 `.md` 文件全文。

编辑模式不按段落、标题、表格单元格拆分编辑，也不提供富文本排版控件。

### 4.3 预览

预览是编辑态下对当前 Markdown 草稿的渲染结果，用于帮助用户确认修改后的阅读效果。

### 4.4 未保存状态

未保存状态表示当前编辑草稿和磁盘上的 Markdown 文件内容不一致。系统需要在 UI 上明确提示，并在高风险离开动作前提醒用户。

### 4.5 文档写回

文档写回是将当前编辑草稿保存到原始 Markdown 文件路径。第三阶段只写回当前打开的本地文件，不做云同步和多端冲突合并。

### 4.6 批注锚点重标记

批注锚点重标记是指 Markdown 正文修改后，系统基于修改后的内容重新定位批注位置。对于 Agent 修改型批注，关联批注需要尽量挂到修改后的目标文本上。

## 5. 功能范围

### 5.1 P0：全篇 Markdown 轻编辑

#### 5.1.1 阅读 / 编辑模式切换

- 页面提供清晰的编辑入口。
- 默认进入阅读模式。
- 用户可以从阅读模式切换到编辑模式。
- 用户可以从编辑模式返回阅读模式。
- 如果存在未保存修改，退出编辑模式前需要提示用户保存或放弃。

#### 5.1.2 整篇 Markdown 源码编辑

- 编辑态展示当前 Markdown 文件全文。
- 用户直接编辑 Markdown 原文。
- 支持编辑 Markdown 中的所有文本内容，包括：
  - 标题
  - 段落
  - 列表
  - 引用
  - 表格源码
  - 普通代码块
  - Mermaid 源码
  - 链接和行内代码
- 编辑器以 textarea 或等价的轻量源码编辑控件为主。
- 第一版不做语法高亮要求。
- 第一版不做块级编辑，不做所见即所得编辑。

#### 5.1.3 编辑态预览

- 编辑模式下支持查看当前草稿的渲染结果。
- 预览使用与阅读模式一致的 Markdown 渲染能力。
- Mermaid、表格、代码块等内容在预览中尽量保持阅读态表现。
- 第一版可以采用“源码 / 预览”切换，不强制左右分栏。

#### 5.1.4 保存到原 Markdown 文件

- 用户点击保存后，将当前草稿写回原 `.md` 文件。
- 保存成功后刷新阅读内容。
- 保存成功后显示 toast。
- 保存失败时展示明确错误信息。
- 保存失败时保留当前编辑草稿，避免用户修改丢失。

#### 5.1.5 未保存状态提示

- 编辑草稿发生变化后，页面显示未保存状态。
- 保存成功后清除未保存状态。
- 放弃修改后清除未保存状态并恢复进入编辑模式时的原文。
- 刷新页面、关闭页面或离开当前文档前，如果存在未保存修改，需要提示用户。

#### 5.1.6 取消编辑 / 放弃修改

- 用户可以取消本次编辑。
- 取消后恢复到进入编辑模式时的 Markdown 内容。
- 如果草稿已发生变化，取消前需要二次提示。
- 取消后回到阅读模式。

#### 5.1.7 保存后批注锚点修复

保存 Markdown 后，系统需要重新解析文档并修复批注锚点。

定位顺序：

1. 优先使用原 heading、块级位置和选中文本定位。
2. 如果原位置失效，使用选中文本和前后上下文做模糊匹配。
3. 如果选中文本发生变化但仍能找到相近段落，尽量挂到相近段落。
4. 如果找不到具体文本，默认挂到原批注所在的上一级标题。
5. 如果上一级标题也找不到，挂到文档顶部的未定位区域，并提示位置可能已变化。

批注锚点修复只改变批注定位信息，不自动改变批注正文、回复内容或状态。

### 5.2 P1：Agent 修改正文工具

#### 5.2.1 MCP 修改正文工具

MCP 增加 `reviewer_apply_document_edit`，允许 Agent 修改当前 Markdown 正文。

工具能力：

- 读取当前 Markdown 内容。
- 根据明确修改意图生成新 Markdown 内容。
- 将修改结果写回原 `.md` 文件。
- 返回修改后的文档信息和保存结果。

产品不要求用户在 UI 里选择“这是修改型批注”。Agent 根据批注内容和用户指令自行判断是否执行正文修改。

#### 5.2.2 支持全文写回

第三阶段技术实现以全文写回为主：

- Agent 可以基于当前全文生成修改后的全文。
- 系统保存修改后的完整 Markdown。
- 不做块级 patch UI。
- 不做复杂 diff 审批流程。

#### 5.2.3 修改后重标记批注锚点

当 Agent 是基于某条批注修改正文时，工具需要支持传入关联批注 id。

修改完成后：

- 系统重新解析修改后的 Markdown。
- 优先将该批注锚点重新标记到修改后的目标文本上。
- 如果目标文本已被大幅改写，则尽量挂到修改后的相关段落。
- 如果相关段落无法判断，则挂到该批注原上一级标题。
- 其他批注也需要按 P0 的锚点修复策略尽量恢复位置。

#### 5.2.4 Agent 回复与状态建议

正文修改完成后，Agent 可以：

- 给关联批注追加回复，说明修改了什么。
- 根据 MCP 工具说明中的建议，视情况将批注标记为 `resolved`。

这些是 MCP 工具使用建议，不是产品侧强制规则。产品 UI 不额外展示“已关联编辑”状态。

## 6. 不支持范围

第三阶段暂不做：

- 块级编辑。
- 富文本编辑器。
- 表格单元格结构化编辑。
- Mermaid 图形化编辑。
- 从批注卡片一键进入对应位置编辑。
- 编辑 diff 预览。
- 保存前复杂审批流。
- 多人实时协作。
- 云同步和账号体系。
- 外部编辑器并发修改的复杂三方 merge。
- 编辑历史、版本快照和恢复。

## 7. 核心流程

### 7.1 用户手动编辑流程

1. 用户打开 Markdown 文档。
2. 系统默认展示阅读模式。
3. 用户点击编辑入口。
4. 系统进入整篇 Markdown 源码编辑模式。
5. 用户修改 Markdown 原文。
6. 系统显示未保存状态。
7. 用户查看预览。
8. 用户点击保存。
9. 系统写回原 Markdown 文件。
10. 系统重新解析文档并修复批注锚点。
11. 系统刷新阅读结果并显示保存成功 toast。

### 7.2 用户放弃编辑流程

1. 用户进入编辑模式。
2. 用户修改 Markdown 原文。
3. 用户点击取消或返回阅读模式。
4. 系统提示存在未保存修改。
5. 用户确认放弃。
6. 系统恢复进入编辑模式时的原文，并回到阅读模式。

### 7.3 Agent 修改型批注流程

1. 用户在文档中留下修改型批注。
2. 用户对 Codex / Agent 说：请处理这份文档里的批注。
3. Agent 通过 MCP 读取文档、批注列表和单条批注上下文。
4. Agent 判断某条批注具有明确修改意图。
5. Agent 调用 `reviewer_apply_document_edit` 修改 Markdown 正文。
6. 系统保存 Markdown，并重新标记关联批注锚点到修改后的文本上。
7. Agent 追加回复说明处理结果。
8. Agent 根据工具建议，视情况将该批注标记为 `resolved`。

## 8. 验收标准

### 8.1 编辑模式

- 用户可以从阅读模式进入编辑模式。
- 编辑模式展示当前 Markdown 文件全文。
- 用户可以修改 Markdown 源码。
- 编辑后能看到未保存状态。
- 用户可以取消编辑并恢复原文。

### 8.2 预览

- 编辑态可以查看当前草稿预览。
- 预览中的标题、段落、列表、表格、代码块和 Mermaid 能正常渲染。
- 预览不会覆盖未保存草稿。

### 8.3 保存

- 点击保存后，原 Markdown 文件被更新。
- 保存成功后阅读模式展示新内容。
- 保存成功后显示 toast。
- 保存失败时显示错误，并保留草稿。

### 8.4 未保存保护

- 存在未保存修改时，退出编辑模式需要提示。
- 存在未保存修改时，刷新或关闭页面需要浏览器级提示。

### 8.5 批注锚点修复

- 保存后，原有批注仍能尽量定位到对应内容。
- 文本轻微修改后，批注能挂到修改后的相近文本。
- 文本找不到时，批注默认挂到上一级标题。
- 批注锚点修复不改变批注状态。

### 8.6 MCP 修改正文

- Agent 可以通过 MCP 修改 Markdown 正文。
- Agent 修改后，原 Markdown 文件被更新。
- 关联批注锚点会重新标记到修改后的文本或相关章节。
- Agent 可以追加回复说明修改结果。
- Agent 可以根据建议将批注标记为 `resolved`。

## 9. 风险与产品取舍

### 9.1 全文编辑风险

整篇 Markdown 源码编辑比块级编辑更直接，但误删和误改风险更高。

第三阶段通过以下方式降低风险：

- 明确未保存状态。
- 支持取消编辑。
- 保存失败保留草稿。
- 离开前提示未保存修改。

### 9.2 批注锚点不可能完全准确

当正文被大幅重写时，批注不一定能精确回到原语义位置。

第三阶段接受“尽量恢复”的产品策略：

- 能定位文本时定位文本。
- 不能定位文本时挂到相关标题。
- 不因为定位失败删除批注。

### 9.3 不做 diff 审批

第三阶段不做编辑 diff 预览和复杂审批。用户手动编辑时以保存前预览作为确认手段；Agent 修改时以批注回复说明修改结果。

后续如果 Agent 大规模修改变多，可以再考虑引入 diff 预览和版本快照。

## 10. 后续预留

- 编辑 diff 预览。
- 版本快照和恢复。
- 外部文件变更检测。
- 查找和替换。
- Markdown 语法高亮。
- 更精细的 Agent patch 工具。
