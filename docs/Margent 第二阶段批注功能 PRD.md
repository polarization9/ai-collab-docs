# Margent 第二阶段批注功能 PRD

## 1. 文档信息

- 产品名称：Margent
- 阶段名称：第二阶段，批注协作闭环
- 文档日期：2026-06-02
- 目标版本：v0.2.0
- 当前状态：需求草案

## 2. 阶段目标

第二阶段从“可阅读的本地 Markdown 审阅器”升级为“可批注、可回复、可被 AI Agent 读取和处理的审阅协作工具”。

目标体验：

> 用户在一份 Markdown 文档里留下若干批注，然后对 Codex / Agent 说：“我在这份文档里做了一些批注，你看看并帮我逐条答复。”Agent 可以通过 MCP 自己读取文档、读取批注、获取批注上下文、追加回复，并在合适时机更新批注状态。

第二阶段重点不做完整编辑器，而是先建立批注数据、批注 UI、批注状态和 MCP 协作能力。对于明确的修改型批注，产品允许 Agent 直接执行文档修改并回写处理结果，不要求产品侧二次确认。

## 3. 用户价值

- 用户可以直接在本地 AI 生成文档上留下问题、修改建议和审阅意见。
- 批注不会散落在聊天记录里，而是和文档一起保存。
- Agent 可以按批注逐条理解上下文并回复，减少用户复制粘贴成本。
- 人工和 AI 都可以推进批注状态，让文档审阅有明确进度。
- 后续轻量编辑和自动修改可以基于批注自然展开。

## 4. 核心概念

### 4.1 批注

批注是用户对文档某个位置留下的审阅记录，包含：

- 被批注位置
- 选中文本或目标对象
- 批注正文
- 回复线程
- 当前状态
- 创建与更新时间

### 4.2 回复线程

每条批注下可以有多条回复。回复可以来自用户，也可以来自 AI Agent。

### 4.3 批注状态

第二阶段只定义两个核心状态：

- `open`：待处理或仍在讨论
- `resolved`：当前认为已经处理完成

产品支持人工和 AI 将批注标记为 `resolved`，也支持重新打开为 `open`。

`resolved` 不等于正文一定被修改，只表示这条批注当前被认为已经处理完。

### 4.4 批注锚点

批注锚点用于把批注重新定位到文档内容。锚点需要支持文档被轻微编辑后的恢复。

## 5. 功能范围

### 5.1 P0：批注闭环 MVP

#### 5.1.1 创建批注

- 用户选中文本后出现添加批注入口。
- 支持对正文段落、标题、列表、引用、代码块和表格文本创建批注。
- Mermaid 第一版支持“整张图批注”，不要求图内精确坐标批注。
- 批注创建后自动出现在右侧批注面板。

#### 5.1.2 批注定位

创建批注时记录：

- 当前文档路径
- 所在 heading id
- 所在 heading 文本
- 块级位置
- 选中文本
- 选区前后上下文
- 字符偏移

点击批注时，页面滚动到对应位置，并短暂高亮目标内容。

#### 5.1.3 右侧批注面板

- 右侧显示当前文档所有批注。
- 默认按文档出现顺序排列。
- 显示批注状态、批注正文、回复数量和最后更新时间。
- 点击批注卡片可以跳转到正文位置。
- 支持收起和展开面板。

#### 5.1.4 批注线程

- 每条批注支持多轮回复。
- 回复区分 `user` 和 `agent`。
- 回复显示创建时间。
- 第一版支持新增回复，不要求编辑历史和删除能力。

#### 5.1.5 状态流转

- 用户可以手动将批注标记为 `resolved`。
- 用户可以将 `resolved` 批注重新打开为 `open`。
- Agent 也可以通过 MCP 更新状态。
- 产品不内置“什么情况下必须 resolved”的硬性判断规则。

#### 5.1.6 本地保存

批注保存到 Markdown 同目录的 `.review.json` 文件。

示例：

```text
context-strategy.md
context-strategy.review.json
```

刷新页面后，批注、回复和状态不丢失。

### 5.2 P1：MCP / Agent 协作能力

#### 5.2.1 MCP 读取文档

MCP 提供读取当前文档内容和元信息的能力，包括：

- 文档绝对路径
- Markdown 原文
- headings
- reviewPath

#### 5.2.2 MCP 读取批注列表

MCP 提供读取批注列表的能力，支持按状态过滤：

- 全部批注
- `open`
- `resolved`

返回内容包括批注正文、状态、锚点摘要、回复数量和更新时间。

#### 5.2.3 MCP 读取单条批注上下文

Agent 处理单条批注时，可以读取：

- 批注正文
- 被批注文本
- 所在章节
- 前后若干段 Markdown
- 相关代码块或表格片段
- 已有回复线程

目标是让 Agent 不只看到一句选中文本，而是拿到足够判断的文档上下文。

#### 5.2.4 MCP 追加回复

Agent 可以给指定批注追加回复。

回复写入 `.review.json`，并在前端刷新或实时同步后显示在右侧面板。

#### 5.2.5 MCP 更新状态

Agent 可以将批注状态更新为：

- `open`
- `resolved`

产品只提供状态更新能力，不把状态切换做成硬规则。

#### 5.2.6 MCP 使用指导

MCP 工具说明中提供建议性行为指导：

- 当 Agent 已完整回答提问型批注时，可以追加回复，并视情况标记为 `resolved`。
- 当 Agent 已按修改型批注完成正文修改时，可以追加处理说明，并视情况标记为 `resolved`。
- 当批注仍需要用户判断、信息不足或只是阶段性讨论时，不建议自动标记为 `resolved`。
- 当用户要求重新讨论某条批注时，Agent 可以将其重新打开为 `open`。

这些是工具使用指导，不是产品侧强制约束。

### 5.3 P1.5：批注锚点修复

当 Markdown 正文发生轻微变化时，系统需要尽量找回批注位置。

定位顺序：

1. 优先使用原始 heading id、块级位置和字符偏移定位。
2. 如果偏移失效，使用选中文本和前后上下文做模糊匹配。
3. 如果选中文本找不到，默认挂到原批注所在的上一级标题下。
4. 如果上一级标题也找不到，挂到文档顶部的未定位区域，并在 UI 中提示位置可能已变化。

批注锚点修复只负责恢复展示位置，不自动改变批注正文或状态。

### 5.4 P1.5：批注驱动修改

用户批注主要有两类意图：

- 提问：用户希望得到解释、判断或建议。
- 修改：用户给出明确修改建议，希望文档被改动。

产品侧不要求用户显式选择批注类型，也不对 Agent 的处理方式做硬性限制。

当 Agent 判断批注是明确修改建议时，可以直接执行文档修改，并在批注线程里写入处理结果。产品不要求二次确认。

文档修改完成后，Agent 可以根据 MCP 使用指导决定是否将批注标记为 `resolved`。

本能力需要文档写入能力配合。第二阶段可先实现批注和 MCP 回复闭环，再接入最小文档写入能力。

## 6. 不支持范围

第二阶段暂不做：

- 批注导出。
- 多人实时协作。
- 云同步和账号体系。
- 完整 Markdown 富文本编辑器。
- Mermaid 图内精确坐标批注。
- 表格单元格结构化编辑。
- 回复编辑历史和复杂审计流。

## 7. 数据结构草案

`.review.json` 示例：

```json
{
  "version": 1,
  "documentPath": "/Users/example/context-strategy.md",
  "updatedAt": "2026-06-02T00:00:00.000Z",
  "annotations": [
    {
      "id": "ann_01",
      "status": "open",
      "createdAt": "2026-06-02T00:00:00.000Z",
      "updatedAt": "2026-06-02T00:00:00.000Z",
      "author": {
        "type": "user",
        "name": "User"
      },
      "body": "这里的策略是不是需要补充失败场景？",
      "anchor": {
        "kind": "text",
        "headingId": "context-strategy",
        "headingText": "Context Strategy",
        "blockIndex": 12,
        "startOffset": 8,
        "endOffset": 32,
        "selectedText": "当前上下文策略",
        "prefix": "在进入模型前，",
        "suffix": "需要进行裁剪和排序"
      },
      "replies": [
        {
          "id": "reply_01",
          "author": {
            "type": "agent",
            "name": "Codex"
          },
          "body": "这里确实可以补充失败场景，我建议增加缓存失效和上下文截断两个分支。",
          "createdAt": "2026-06-02T00:01:00.000Z"
        }
      ]
    }
  ]
}
```

## 8. 前端交互

### 8.1 创建批注

用户选中文本后，选区附近出现添加批注按钮。点击后打开轻量输入框，提交后创建批注。

### 8.2 正文高亮

有批注的文本显示轻量高亮。高亮不应影响正文阅读，也不应破坏代码块和表格布局。

### 8.3 右侧面板

右侧面板展示批注列表和当前选中批注详情。用户可以在面板中回复、解决或重新打开批注。

### 8.4 状态筛选

支持筛选：

- 全部
- 未解决
- 已解决

默认显示未解决和当前选中批注。

## 9. 服务端接口

第二阶段本地服务需要新增接口：

### GET `/api/review`

读取当前文档对应的 `.review.json`。如果文件不存在，返回空批注结构。

### PUT `/api/review`

保存完整 review 数据。

### POST `/api/review/annotations`

创建批注。

### POST `/api/review/annotations/:id/replies`

给批注追加回复。

### PATCH `/api/review/annotations/:id/status`

更新批注状态。

### GET `/api/review/annotations/:id/context`

读取单条批注上下文，供前端和 MCP 复用。

## 10. MCP 工具草案

### `reviewer_get_document`

读取当前文档内容和元信息。

### `reviewer_list_annotations`

读取批注列表，支持按状态过滤。

### `reviewer_get_annotation_context`

读取指定批注及其上下文。

### `reviewer_add_annotation_reply`

给指定批注追加回复。

### `reviewer_update_annotation_status`

更新批注状态为 `open` 或 `resolved`。

### `reviewer_apply_document_edit`

根据明确修改建议更新 Markdown 文档。该工具可在批注闭环稳定后实现。

## 11. 验收标准

- 用户可以选中文本创建批注。
- 批注刷新后仍保存在 `.review.json`。
- 右侧面板可以展示所有批注。
- 点击批注可以跳转到正文位置。
- 用户可以给批注追加回复。
- 用户可以将批注标记为 `resolved`，也可以重新打开为 `open`。
- MCP 可以读取批注列表。
- MCP 可以读取单条批注上下文。
- MCP 可以给批注追加 Agent 回复。
- MCP 可以更新批注状态。
- 文档轻微修改后，批注位置能优先恢复；找不到原文时挂到上一级标题下。

## 12. 推荐实现顺序

1. `.review.json` 数据结构与读写接口。
2. 前端创建批注和右侧批注面板。
3. 批注回复线程和状态流转。
4. 正文高亮、点击跳转和筛选。
5. 批注锚点修复。
6. MCP 读取批注、读取上下文、追加回复、更新状态。
7. 明确修改型批注的文档写入能力。
