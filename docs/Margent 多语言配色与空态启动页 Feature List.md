# Margent 多语言、配色与空态启动页 Feature List

## 1. 文档信息

- 产品名称：Margent
- 模块名称：基础体验增强
- 文档日期：2026-06-05
- 当前状态：Feature List 草案
- 覆盖范围：多语言、配色方案、空态启动页、macOS 菜单栏、文档内操作区、统一设置入口、批注锚点稳定性

## 2. 模块目标

这组功能属于 Margent 的基础体验层，不改变文档阅读、批注、编辑、MCP / Codex 连接的核心流程，主要解决五个问题：

- 用户能用中文或英文理解产品界面。
- 用户能选择更适合自己阅读习惯的界面配色。
- App 在没有打开文档时，不再只是一个空白入口，而是给用户清晰、轻量、可操作的启动页。
- 语言、配色、启动行为、连接状态等设置有统一入口；App 级入口优先放在 macOS 原生菜单栏，不分散在空态、阅读页和局部工具栏里。
- 文档被用户或 Codex 修改后，历史批注的位置尽量跟随正文变化，不因为旧 block id 漂移而跳到不相关内容上。

## 3. 功能清单

### 3.1 多语言支持：中文 / 英文

#### P0

- 支持中文和英文两套界面文案。
- 首次启动默认跟随系统语言：
  - 系统语言为中文时，默认中文。
  - 其他语言环境下，默认英文。
- 支持用户在 App 设置面板中手动切换语言，设置入口从 macOS 菜单栏进入。
- 语言选择需要持久化，下次启动保持用户选择。
- 覆盖现有主要 UI 文案：
  - 空态启动页
  - 文档内操作区
  - 目录栏
  - 批注列表
  - 批注创建 / 回复 / 删除 / 状态操作
  - 编辑态保存 / 退出
  - Mermaid 工具按钮与 toast
  - 文件打开、加载、错误状态
  - macOS 原生菜单栏中的 App 级菜单项
- 不翻译用户打开的 Markdown 正文内容。

#### P1

- 根据语言切换时间、日期、数量等格式化展示。
- 对 MCP / Codex 连接相关提示补齐英文文案。
- 为新增功能建立文案 key 命名规范，避免后续文案散落在组件中。

#### 暂不考虑

- 机器翻译文档内容。
- 多语言同时显示。
- 除中文、英文之外的第三语言。

### 3.2 配色方案：默认 / 蓝白 / 灰白

#### P0

- 在现有默认配色之外，新增两套完整配色方案：
  - 蓝白：偏清爽、理性、文档工具感。
  - 灰白：偏中性、低干扰、长时间阅读友好。
- 配色切换需要影响完整 UI token，而不只是页面背景：
  - 页面背景
  - 正文区域
  - 目录栏
  - 批注列表
  - 按钮、输入框、分段控件
  - toast
  - 表格
  - 代码块
  - Mermaid 容器与工具栏
  - 选区、批注高亮、hover / active / disabled 状态
- 支持用户在 App 设置面板中手动切换配色方案，设置入口从 macOS 菜单栏进入。
- 配色选择需要持久化，下次启动保持用户选择。
- 默认配色仍作为第一套方案保留。

#### P1

- 支持跟随系统浅色 / 深色模式，但不作为独立的用户可见配色状态。
- 为不同配色补齐 Mermaid lightbox、代码编辑器、表格列宽调整等复杂组件状态。
- 建立配色 token 表，避免组件内硬编码颜色继续扩散。

#### 暂不考虑

- 用户自定义色值。
- 主题市场。
- 每个文档单独保存配色。

### 3.3 空态启动页设计

#### P0

- 当 App 没有打开任何文档时，展示启动页。
- 启动页保持简洁，不做营销页，不出现大段产品介绍。
- 核心操作：
  - 打开 Markdown 文件
  - 查看最近打开文件
- 打开文件也需要作为 macOS 菜单栏中的 App 级入口存在，空态按钮只是快捷入口。
- 最近打开文件展示：
  - 文件名
  - 文件路径
  - 最近打开时间
  - 文件不存在时给出轻量状态，不阻断页面
- 启动页需要体现 Margent 的产品气质：
  - 文档工具感
  - 安静、清晰、低干扰
  - 不使用营销式 hero、过度装饰或大面积渐变
- 支持空态下进入统一 App 设置面板，但打开的是同一个设置面板。
- 文件打开失败时，在启动页内显示可读错误提示，并允许重新选择文件。

#### P1

- 支持将 Markdown 文件拖入启动页打开。
- 支持展示当前 Codex / MCP 连接状态，但不抢占主操作。
- 支持从启动页恢复上次关闭前的文档。
- 支持清理最近文件记录。

#### 暂不考虑

- 文档模板库。
- 云端文件入口。
- 账户系统。
- 团队协作入口。

### 3.4 macOS 菜单栏与文档内操作区

#### P0

- 使用 macOS 原生菜单栏承载 App 级命令，不在页面内自建一条重复的 App 顶部栏。
- macOS 菜单栏至少包含：
  - 打开 Markdown 文件
  - 最近打开文件
  - App 设置
  - 退出 App
- App 设置面板从 macOS 菜单栏进入，用于承载语言、配色方案、启动行为、最近文件管理等全局偏好。
- 文档内只保留当前文档上下文相关的高频操作：
  - 当前文档路径 / 文件名
  - 编辑 / 保存 / 退出编辑
  - 批注列表入口
  - 当前文档的 Codex 连接和自动监控状态
- 阅读态、编辑态、空态都复用同一套 App 设置面板，不在各自页面重复定义设置入口。
- 需要显式配置 macOS 原生菜单项和文案，不继续使用 Tauri / WebView 默认菜单暴露出的无关项。
- 文档内操作区不承载低频解释文本，状态说明进入 tooltip、toast 或设置面板内部。

#### P1

- 支持 macOS 菜单栏快捷键：
  - 打开文件：Cmd+O
  - 打开设置：Cmd+,
  - 保存：Cmd+S，仅编辑态可用
- 支持 macOS 菜单栏根据当前状态启用 / 禁用菜单项，例如没有打开文档时禁用保存。
- 支持最近打开文件作为 macOS 菜单栏子菜单。
- 支持在设置面板内查看当前 App 版本、MCP 连接状态、默认 Markdown 打开方式状态。

#### 暂不考虑

- 多级页面导航。
- 账户、团队、云同步入口。
- 将文档内编辑工具全部搬进 macOS 菜单栏。

### 3.5 批注锚点稳定性

#### P0

- 保存 Markdown 时，系统需要同时读取编辑前正文和编辑后正文，建立 old block -> new block 的映射关系。
- 每次文档保存后，回扫当前文档的全部历史批注，并基于 block 映射更新批注锚点：
  - `blockId`
  - `blockIndex`
  - `headingId`
  - `headingText`
  - 文本锚点的 `startOffset` / `endOffset`
- block 映射优先级：
  - 内容完全相同的 block 直接映射。
  - 同一标题范围内、类型一致、顺序接近的 block 优先映射。
  - 标题重命名但位置连续时，识别为同一章节 block 的演进，例如从“应用顶部栏与统一设置入口”改为“macOS 菜单栏与文档内操作区”。
  - 通过编辑前后 diff 将旧 block 的字符区间映射到新文档区间。
- 如果选中文本仍存在，更新到精确文本位置。
- 如果选中文本已经被改写，但 block 或标题仍可信，批注降级挂到对应 block 或上一级标题。
- 如果 block 被删除、拆分、合并且无法可靠判断，批注不删除，挂到原上一级标题，并标记为位置不精确。
- 前端定位批注时，不能只因为旧 `blockId` 在当前页面中存在就直接定位；必须校验文本、上下文或锚点可信度。
- 当前锚点校验失败时，前端需要走文本匹配、标题 fallback 或“不精确位置”提示，避免跳到不相关内容。

#### P1

- 为 anchor repair 建立最小单元测试，覆盖标题重命名、段落新增、段落移动、block 删除、block 拆分和 block 合并。
- 在批注卡片或正文高亮上展示轻量状态，提示“位置可能已变化”，但不干扰正常阅读。
- 对 Agent 修改型批注，MCP 工具继续支持传入 `preferredSelectedText`，并把该批注优先锚定到修改后的目标文本。

#### 暂不考虑

- 语义级向量匹配。
- 大段重写后的自动语义恢复。
- 多人同时编辑下的复杂冲突合并。

## 4. 推荐优先级

建议开发顺序：

1. 先确定 macOS 原生菜单栏、文档内操作区和统一设置面板骨架，因为语言、配色、空态入口都依赖它。
2. 并行修复批注锚点稳定性，优先解决旧 `blockId` 漂移导致的错误定位。
3. 再做空态启动页基础结构，让没有文档时也能进入打开文件、最近文件和设置。
4. 再做配色 token 化和三套配色方案。
5. 最后做多语言文案抽取，因为它会触达大量组件，适合在 UI 结构稳定后统一替换。

## 5. 待确认问题

- macOS 菜单栏中需要放哪些 App 级入口：打开文件、最近文件、设置、语言 / 配色，哪些仍保留在文档内操作区？
- 设置面板中的“当前文档”分组是否只展示 Codex 连接和自动监控，还是也放入阅读偏好、Mermaid 展示偏好等文档级设置？
- 配色方案是否只在浅色体系内变化，还是默认方案继续保留当前的 light / dark 切换能力？
- 最近打开文件是否需要跨 CLI / 桌面 App 共享，还是只记录桌面 App 打开的文件？
- 空态启动页是否需要展示“打开示例文档”入口？
- 批注位置不精确时，是只在批注卡片展示提示，还是正文高亮也要降级为章节级提示？
